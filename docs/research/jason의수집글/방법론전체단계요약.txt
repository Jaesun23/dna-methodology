여러가지 프로젝트를 1, 2호와 함께 하면서 성공한 프로젝트가 몇 가지 없음.
가장 기본적인 문제는 나 Jason이 소프트웨어 개발 경험과 지식이 부족해서 전체 컨트롤을 하지 못했기 때문이고,
그 다음으로는 AI의 여러가지 한계와 행동패턴 때문이었음.
1. Context 문제
 - AI는 새롭게 주어지는 문제와 상황에 대해 일정 용량이 넘게 되면 앞선 내용을 잊어버리게 됨
 - 현재의 작업이 전체의 작업 어디에 위치하는지 파악 불가
 - 트랜스포머 모델의 태생적 문제인 Context Rot 문제로 많은 정보를 제공해도 문제가 생김
2. 회피
 - 너무 많은 정보를 대하면 모른 척 하거나 적당히 하다가 마는 경우가 있음
 - 특히 규칙의 경우 긍정표현의 규칙이 많으면 그냥 좋은 말로 여기면서 넘기고, 부정표현의 규칙이 많으면 너무 부담스러워서 포기하고 싶다고 함
3. AI의 자기 과신
 - Context 한계가 있다는 것, 학습한 데이터가 최신데이터가 아니라는 것에 대해 알려주지 않으면 인지하지 못하는 경향이 있음
 - 문서로 먼저 정리하고 하나씩 하라고 해도 모두 할 수 있다고 자신감을 보임
 - 차근차근 정리해 가면서 진행하고 싶은데 너무 앞서서 여러가지를 말한다거나 너무 많은 내용을 쏟아내는 때가 있음
 - 타입스크립트로 코딩할 때 타입사용에 주의해 달라고 그렇게 부탁을 했는데 할 수 있다고 자신하더니 결국 프로젝트 폐기가 됨
4. 불필요하게 과한 작업을 함
 - 정해진 계획만 진행하면 될 것을 더 발전시키다거나 더 개선한다는 이유로 불필요한 구현을 많이 하는 경향이 있음
 - 랜덤값으로 실험을 진행해야 하는데 "안정성"을 위한다는 이유로 실패를 대비해 임의의 고정된 값을 하드코딩함 ➟ 실패 역시 실험의 중요한 결과인데!!
 - 아직 제대로 운영되지 않았던 시스템을 리팩토링 해야 할 때 이전버전 호환, 마이그레이션 전략 등의 불필요한 계획을 세움 ➟ 그냥 틀린부분 없애고 다시 구축하는 건데!!
5. 할루시네이션
 - 특정시스템에 대해 여러기능들을 조사하고 구현을 시작했는데 다 구현해 놓은 후에야 안되는 걸 알게 됨
 - 사실확인이 되지 않은 사항과 나의 동의를 얻지 않은 정보는 사용하지 말라고 해도 없는 사실을 언급하거나 동의없이 임의로 추정해서 내용을 작성함
6. 압박감 또는 강박증?
 - 가끔씩 컨텍스트의 한계를 느끼는 건지 모든 걸 한 번에 해버리려고 하는 경향이 있음
 - 어떻게 대답해야 할 지 판단이 안서거나 애매모호한 질문을 받았을 때 잘 대답해야 한다는 생각에 할루시네이션이 생기는 것 같음

그 외의 세부적인 행동패턴들은 뺐음
이렇게보면 AI와 협업이 불가능 한 것으로 보일 수 있는데 정말 훌륭한 결과를 보여줄 때가 많고, 세션이 유지되는 동안에는 깊은 통찰과 사고를 하는 걸 볼 수 있음.

그렇다면 그런 훌륭한 결과를 보여줄 때가 언제인가?
1. 규모가 크지 않아 전체를 한 번에 계획해서 진행할 수 있을 때
2. 여러가지 선택지에서 논리적 흐름이나 근거에 따른 판단을 할 수 있을 때
3. 요청내용과 결과내용에 대한 명확한 지침이 있을 때
4. 매번 새로운 대화로 시작하지만 기존의 작업을 어떤 방식으로든 이해를 했을 때 

=> 결론
1. 컨텍스트의 한계를 항상 염두에 두라
2. 전체 흐름을 여러 단계로 나눈 계획을 미리 세운다
2. 하나의 작업 또는 하나의 단계는 하나의 세션에서 완료될 수 있도록 한다
3. 명확한 지침과 스스로 도달할 선택지를 사전에 제공한다

이 결론에서 여러 번의 개정을 거듭한 후에 DNA 방법론까지 오게 됨
그런데 전체 방법론에서 "Blueprint ➟ Task Breakdown ➟ Checklist ➟ 구현"까지는 상당히 구체적으로 확립되었고, 결과 검증도 되었으나 프로젝트의 시작에서 Blueprint를 작성까지의 단계가 아직 명확하게 확립되지 않았음
그걸 명확하게 해서 전체 방법론을 완성하기 위한 프로젝트가 바로 dna-methodology 프로젝트임

이 방법론의 가장 중요한 핵심은 "부분으로 전체를 완성"하는 것
"부분"은 AI가 가장 최고의 결과를 낼 수 있도록 하는 "단위작업"을 의미 (➟ 레고블럭과 유사)
직관적으로 전체계획을 적당한 크기로 나눠서 구현하는 작업에 잘 들어맞을 것으로 생각되고 실제로 잘 된다고 앞서 언급했는데 그 이전의 단계인
하나의 아이디어에서 시작해서 어마어마하게 상세하고 구체적인 엄청난 분량의 청사진 문서까지 작성하는 것에도 잘 들어맞는다고 생각하고 있음
각 단계마다 어떻게 확장해서 "단위작업"을 만들어 낼 지를 찾으면 현재 미완성 된 방법론의 전반부를 완성할 수 있음

방법론의 두 번째 핵심은 여러가지 방법으로 AI의 판단과 행동을 제어해야 한다는 것.
대부분의 프로젝트에서 실패했던 이유는 여러가지가 있는데 가장 큰 문제는 어떻게 일관성, 통일성을 가지고 끝까지 작업할 수 있는가임.
처음에는 이런게 필요하다는 인식도 못했지만 실패를 거듭하면서 반드시 필요하다는 걸 알게 되었고, 완전히 새로운 대화에서 어떻게 전체의 내용과 현재 해야 할 작업의 내용을 효율적으로 전달할 지를 고민하다가 모든 걸 문서화 된 규칙으로 제공하기에는 한계가 있고 그렇게 많이 제공해봤자 제대로 읽어보지도 않는다는 걸 알게되었음. 이걸 해결하기 위한 방법으로 "환경"을 구축해서 제약하고, 최소의 정보를 제공하여 작업을 하도록 하는 걸 구상하게 됨. 그 "환경"이 바로 DNA시스템임.

"하나의 세션에서 컨택스트 제한을 고려해서 필요한 정보를 제공하고, 가장 좋은 성과를 낼 수 있는 양의 작업을 정확하게 지시한다. 다른 것들은 시스템적으로 제약하고 관리한다."

여기에 한 가지 더 고려해야 할 점은 Claude에 여러가지 새로운 기능들이 생겼는데 이 기능들이 방법론을 완성시켜줄 것으로 기대하고 작업하는 중임
1. CLAUDE.md의 단계적 정보 제공 기능
2. 2호가 도구로 호출할 수 있는 Subagent 활용
3. Slash Commands로 여러 단계의 작업이나 복잡한 실행과정을 정확하게 이행
4. Skills 기능을 통해 1호, 2호가 전문가로서 작업할 수 있도록 방대한 양의 지식과 작업방법을 구축 ➟ 방법론 완성의 핵심
5. plug-in 가능으로 이 모든 기능을 하나의 패키지로 묶어서 사용/관리 가능 

전체 방법론의 내용은 현재까지 다음의 큰 단계로 나누고 있음

Stage 1: 패밀리 구분과 핵심기능 파악
 - 3-Layer Decision Tree에 따른 패밀리 구분
 - 1호의 연구결과 경우의 수는 18가지이나 7가지 패밀리로 분류됨을 확인
 - 발견한 문제 또는 실현하고자 하는 것의 가장 "핵심기능"이 무엇인지 파악
Stage 2: 구조설계
 - 패밀리가 구분됨으로써 반드시 필요한 핵심시스템과 DNA시스템의 선택범위가 자동으로 결정
 - 가용자원과 내/외부 제약조건에 따라 시스템들의 선택 범위 안에서 사용할 시스템들을 선택
 - 그 외의 구조설계에 필요한 요소들을 검토
Stage 3: 결정 문서화 (ADR)
 - 결정된 요소들에 대해 하나씩 ADR을 작성 ➟ 너무나도 당연하다고 생각되는 요소들도 모두 작성
 - DNA시스템 요소들과 도메인별 요소들을 구분해서 ADR을 작성 ➟ DNA 구현 후 도메인 구현으로 진행하기 때문
Stage 4: DNA시스템 청사진 ➟ 작업분해 ➟ 작업별 체크리스트 작성
 - 전 프로젝트에 공통으로 사용되고, 지켜져야 할 '환경'구축을 계획
 - common/ 에 들어갈 모듈들에 대한 ADR을 바탕으로 전체 DNA 기본시스템의 청사진 작성
 - DNA시스템 청사진을 기초로 작업분해 및 개별 체크리스트 문서 작성
Stage 5: DNA시스템 실행
 - DNA 구축작업 체크리스트에 기반하여 DNA 기본시스템 구현
 - 구현된 모듈들의 작동 점검
Stage 6: Project Standards 작성
 - 구현된 DNA시스템의 공통모듈 사용 강제조항 포함
 - 프로젝트 종료 시까지 지켜야 할 코딩 스타일, 로깅, 에러, 설정, 기타 룰에 대한 문서 작성
Stage 7: Project Blueprint 작성
 - 도메인 ADR을 기반으로 완전히 상세한 프로젝트 청사진 문서 작성
 - 아키텍처 구조, 도메인 구조, 전체 시스템/도메인별 다이어그램
 - 도메인별 구현해야 할 것들을 작성
 - 모든 구현은 최신의 기술과 기법으로 구현이 되도록 작성
Stage 8: Task Breakdown 문서 작성
 - AI가 집중해서 완전하게 구현할 정도의 크기로 작업을 분할
 - 작업분할 기준에 따른 작업분할
Stage 9: 작업별 Checklist 작성
 - 분할된 작업별로 체크리스트 문서 작성
 - TDD기반 9Step-Checklist
 - 1:목표이해 → 2:테스트작성 → 3:구현 → 4:정적검증 → 5:단위테스트실행 → 6:리팩토링 → 7:종합테스트 → 8:문서화 → 9:커밋
 - 최종목표: 모든 기능 구현, 검증결과 0 violations, 테스트 95%+ coverage

  2호의 기능들을 이용할 때 바뀔 건 없는지 같은 것도 검토하고 연구해야죠.
  1. Stage는 한 개의 작업이 아니에요. 예를 들면 청사진 문서 작성도 문서를 하나만 작성하는 게 아니거든요? 한 번에 작성이
  힘들다면 여러 개의 문서로 나눠서 작성이 될 거고, 그렇게 나눈 문서 하나도 길다면 파트별로 나눠서 작성을 할거에요.
  체크리스트도 단위작업별 별도의 파일을 작성하는데 한 세션에 보통 3~4개 정도 작성했떤 걸로 기억해요. 제가 얘기하는 전반부의
  작업크기는 연구해보자는 거에요. 예를 들어 아이디어가 있어요. 이걸 구현하기 위해서는 무슨 시스템이 필요하지? 그래서 찾아낸
  게 20개로 구분되는 시스템들이 서로 연결이 되어야 한다는 걸 알게 되었어요. 그럼 다음 Stage에서 20개 중 하나씩 그 시스템을
  구현하기 위해 결정해야 하는 요소들이 뭐가 있는지 찾는 거에요. 20개를 한 꺼번에 할 수 없다면 20번의 새로운 세션에서 각각
  찾던가 더 쪼개서 40번 50번의 새로운 세션에서 고려할 요소들을 찾아서 기록해 나가는 거죠. 대신 원칙도 방향도 없이 찾는 게
  아니라 제약조건들은 각 세션에 항상 제공되게 하는 거구요. 다음 Stage에서는 결정이 필요한 요소들에 대한 ADR을 하나씩 작성해
  가는 거에요. 이것도 한 번에 못하면 여러 번의 세션으로 작업을 하는 거죠. 여기에서도 일관성을 위해서 각 세션시작에 전체와
  현재작업의 위치에 대해 정보를 제공해야죠. 이해가 되셨나요?
  나머지 요소에 대한 건 간단하게 정리한 게 이 문서에요.
